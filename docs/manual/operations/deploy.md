# Operations Runbook: Deploy a Production Release

**Applies to:** udr 0.1.0 · **Revised:** 2026-06-08

## Scope

This runbook covers building a self-contained production [relx](../glossary.md) release of the `udr` HSS/UDR, placing its configuration, and starting it on a target host. It is for operators deploying the node for the first time on a host. Starting, stopping, and attaching a console to an already-deployed release are covered in [`RUN-LIFECYCLE-001`](lifecycle.md); in-place replacement of a deployed version is covered in [`RUN-UPGRADE-001`](upgrade.md).

---

## RUN-DEPLOY-001: Build and deploy a production release

### Purpose

*(Informative.)* This procedure produces a `prod`-mode release — a self-contained directory that bundles the application code, the Erlang runtime system, `config/sys.config`, and `config/vm.args` — and starts it as a running HSS node. An operator runs it to bring a new node into service.

### Pre-conditions

*(Normative.)* The following `shall` hold before starting:

- [Erlang/OTP](../glossary.md) 29 or later and [rebar3](../glossary.md) are installed on the build host, per [install.md](../install.md) §1.
- A working copy of the source has been obtained and `rebar3 compile` has succeeded on it ([install.md](../install.md) §3).
- The intended configuration values are known: the data backend (ETS or MongoDB; see the [data-store configuration reference](../configuration/data-store.md)), the [S6a](../glossary.md) identity and `listen` address (see the [S6a Diameter configuration reference](../configuration/diameter.md)), the listener bind addresses, the node name, and the distribution cookie (see the [node reference](../configuration/node.md)).
- Where the release is run on a host other than the build host, that host has a compatible operating system and architecture, because a `prod`-mode release bundles the Erlang runtime system.

### Inputs and privileges

- The source tree and the target deployment directory.
- Permission to bind the configured listener ports on the target host.
- The finalized `config/sys.config` and `config/vm.args`.

> [!CAUTION]
> The shipped `config/vm.args` sets the well-known distribution cookie `udr_cookie` and the shipped `config/sys.config` binds every listener to loopback. Before exposing a node to any network, the cookie `shall` be replaced and the bind addresses reviewed, per the [node reference](../configuration/node.md) §5.2 and the [S6a Diameter configuration reference](../configuration/diameter.md) §5.3. Secret material handling is covered in [`RUN-SECRETS-001`](secrets.md).

### Steps

1. Set the configuration before building, so it is bundled into the release. Edit `config/sys.config` for the backend, S6a identity, and listener addresses, and `config/vm.args` for the node name and cookie. The relx section reads these two files by path (confirmed in `rebar.config`: `{sys_config, "./config/sys.config"}`, `{vm_args, "./config/vm.args"}`).
2. Build the production release from the project root:

   ```sh
   rebar3 as prod release
   ```

   This assembles release `udr` version `0.1.0` in `prod` mode under `_build/prod/rel/udr` (the release name and version are set in the `relx` section of `rebar.config`; the `prod` profile overrides `mode` to `prod`).
3. Where the target host differs from the build host, package and copy the release directory `_build/prod/rel/udr` to the target host (for example with `tar` and `scp`), preserving its layout.
4. Start the node in the background on the target host:

   ```sh
   _build/prod/rel/udr/bin/udr daemon
   ```

   To run the node in the foreground with an attached console instead, use `_build/prod/rel/udr/bin/udr console` (see [`RUN-LIFECYCLE-001`](lifecycle.md)).

> [!NOTE]
> `bin/udr` is the release control script generated by relx for the `udr` release. The subcommands referenced here (`daemon`, `console`, `ping`, `remote_console`, `stop`) are the standard relx release-script subcommands; confirm the exact set on the target with `_build/prod/rel/udr/bin/udr` run without arguments, which prints the usage.

### Verify

*(Observable outcome.)*

- Confirm the node is up:

  ```sh
  _build/prod/rel/udr/bin/udr ping
  ```

  The response `shall` be `pong`.

- Confirm the provisioning listener answers. From the target host (or any host that can reach the configured provisioning address):

  ```sh
  curl -s -o /dev/null -w '%{http_code}\n' \
    http://127.0.0.1:8090/provision/v1/subscribers/001010000000001
  ```

  A reachable node with no such subscriber returns `404`, which confirms the listener is bound and serving.

- Confirm the S6a listener is bound on the configured address and port:

  ```sh
  ss -ltn '( sport = :3868 )'
  ```

  A socket `shall` be listed on the configured address (`127.0.0.1` with the shipped default, or the routable address set in `listen`).

### Rollback / on failure

- If `bin/udr ping` returns anything other than `pong`, the node did not start. Inspect the release log under `_build/prod/rel/udr/log/` (or the path the release script reports) for the boot error, correct the configuration, and repeat from Step 2.
- If a listener `Verify` fails, confirm the bind address and port in `config/sys.config` and that no other process holds the port (`ss -ltn`), then restart the node (see [`RUN-LIFECYCLE-001`](lifecycle.md)).
- To return the host to its prior state, stop the node (`_build/prod/rel/udr/bin/udr stop`) and remove the copied release directory.
